# Mini PRD: BwAI in Kwansai 2026 Network

## Overview / Problem Statement

BwAI in Kwansai 2026 の会場（IPUT）において、参加者・運営が安定してインターネットに接続できるネットワークを構築する。

2026-03-24 の下見時に会場ネットワークの事前検証を実施したところ、**事前に「プロキシ解除済み」との報告を受けていたにも関わらず、プロキシが有効のまま残っており一切の通信ができなかった。** パケットキャプチャ解析の結果、以下が確認されている：

- `proxy.pac` が配信中（デフォルト `return PROXY;`）
- `proxy01.osaka.hal.ac.jp:8080` が HTTP 407（Basic 認証要求）を返却
- DIRECT_LIST 外ドメインの直接通信がファイアウォールで遮断
- ICMP、WireGuard（UDP 51820）等のプロキシ非対応通信も全て遮断

会場側ネットワークのプロキシ解除が当日も確実に行われる保証がないため、**会場側設備に依存しない自前のネットワークインフラ**を構築し、確実に通信可能な環境を提供する。

## Goals

- イベント参加者・運営スタッフ・配信担当が安定してインターネットに接続できる
- 会場側プロキシ環境の状態に左右されない通信経路を確保する
- VLAN によるネットワーク分離で運営トラフィックと参加者トラフィックを隔離する
- 50〜100 台規模のクライアント接続に耐えうる WiFi 環境を提供する

## Non-Goals

- 会場側の既存ネットワーク機器の設定変更（管理外のため）
- BGP による経路冗長化（オプションとして検討はするが必須要件ではない）
- 完全な耐障害性（商用レベルの SLA は求めない）
- Captive Portal / 802.1X による本人確認（参加者はエンジニア中心でリテラシーが高く、AUP + 通信ログで十分）
- ランダム MAC アドレスの無効化要求（per-SSID 固定のためイベント内の追跡には影響なし）

## User Stories

| ユーザー | ストーリー |
|---|---|
| 参加者 | WiFi に接続し、ブラウザ・ターミナル・各種開発ツールから制限なくインターネットにアクセスしたい |
| 運営スタッフ | 参加者と分離されたネットワークで、運営用ツール・管理画面に安全にアクセスしたい |
| 配信担当 | YouTube Live 等への安定したアップリンクを確保し、参加者トラフィックの影響を受けずに配信したい |
| ネットワーク管理者 | Grafana / SNMP でリアルタイムにネットワーク状態を監視し、障害発生時に迅速に対応したい |

## Proposed Solution

### アーキテクチャ概要

会場の有線ポートをアップリンクとして利用し、自前の VyOS ルーターから VPN トンネルで自宅ルーターまたは GCP 経由でインターネットに抜ける構成を取る。これにより会場側プロキシを完全にバイパスする。

```
[The Internet]
    │
    ├── [自宅 Router / NAPT (AS65002)] ◄══ backup BGP ══╗
    │                                                     ║
    ├── [GCP VGW Site-to-Site VPN (AS64512)] ◄══ primary BGP ══╗
    │                                                           ║
    └── [会場設備 (管理外 blackbox)] ── DHCP/NAPT ── [VyOS Router (AS65001)]
                                                        │
                                              ┌─────────┴─────────┐
                                          [PoE Switch]      [Local Server]
                                              │              DNS / DHCP / Grafana
                                    ┌─────────┼─────────┐
                                [Cisco AP]  [配信PC]  [スピーカー]
                                 vlan 11,30,40  vlan 20   vlan 30
```

※ 詳細な構成図は [`BwAI.md`](../../BwAI.md) を参照

### VPN 戦略（プロキシ状態に応じた二段構え）

| 条件 | VPN 方式 | 理由 |
|---|---|---|
| プロキシ無効化済み | WireGuard or IKEv2 | UDP ベースで高速・低遅延。下見時は UDP 51820 が遮断されていたため、プロキシ解除が前提 |
| プロキシ無効化不可 | SoftEther VPN | HTTPS (TCP 443) 上でトンネルを張れるため、プロキシの CONNECT メソッド経由で通過可能 |

### VLAN 設計

| VLAN ID | 名称 | サブネット | 用途 |
|---|---|---|---|
| 11 | mgmt | 192.168.11.0/24 | 管理系（DNS, DHCP, Grafana, SNMP Exporter, Ansible） |
| 20 | live | 192.168.20.0/24 | 配信用（YouTube Live Streaming） |
| 30 | staff | 192.168.30.0/24 | 運営スタッフ |
| 40 | user | 192.168.40.0/22 | 参加者（/22 で最大約 1,000 アドレス） |

### ACL（アクセス制御）

- `vlan 40 (user) → vlan 11`: DNS (UDP/TCP 53), DHCP (UDP 67-68) のみ許可
- `vlan 40 (user) → vlan 11`: Grafana, SNMP Exporter, Ansible 等は拒否
- `vlan 40 (user) → 0.0.0.0/0`: 許可（デフォルトルート）
- `vlan 40 (user) → 他 vlan`: 拒否
- 他 vlan 間: 相互通信許容

### WiFi

| 選択肢 | 詳細 | 判定 |
|---|---|---|
| **会場 Aruba AP + WLC（推奨）** | 既設の Aruba AP で VLAN を切り、WLC からイベント用臨時 SSID を配信。会場側の協力が必要 | 会場側と要交渉 |
| **自前 Cisco Aironet 3800** | 20 台程度を用意可能。PoE スイッチ経由で展開 | 準備中（バックアッププラン） |
| **Buffalo 家庭用ルーター ×10 カスケード（会場側提案）** | 家庭用 WLC は 10 台程度が限界。50〜100 台のクライアント管理は不可能。EasyMesh は 4 台まで。メッシュの air-to-air はさらに不安定 | **却下** |

### 監視・運用

| コンポーネント | 配置 | 役割 |
|---|---|---|
| DNS forwarding | VyOS (r3) | 名前解決 (PowerDNS Recursor) |
| DHCP / DHCPv6 | VyOS (r3) | IP アドレス配布 (内部 Kea) |
| Grafana | ローカルサーバ (local) / GCE (active, 外部公開) | ネットワーク監視ダッシュボード |
| Prometheus SNMP Exporter | ローカルサーバ | スイッチ・AP の SNMP メトリクス収集 |
| rsyslog | ローカルサーバ → GCE (GCS 転送) | ログ集約 |
| NetFlow v9 | VyOS (r3) → nfcapd (ローカルサーバ) | 通信フロー記録 (法執行機関対応) |
| DNS クエリログ | VyOS (r3) → rsyslog | DNS 照会記録 (法執行機関対応) |
| DHCP forensic log | VyOS (r3) Kea hook → rsyslog | IP ↔ MAC ↔ hostname 紐付け記録 |
| NDP テーブルダンプ | VyOS (r3) cron 1 分 → rsyslog | IPv6 ↔ MAC 記録 (iOS/Android 対応) |

※ DNS/DHCP は VyOS に統合。アップリンクが 1 本のため GCE 冗長化は廃止 (r3 障害時 GCE に到達不能)。通信ログの詳細は [`logging-compliance.md`](../design/logging-compliance.md) を参照。

## High-Level Tech Consideration

### ルーター

- **VyOS** を採用。VPN（WireGuard / SoftEther）、VLAN、ファイアウォール、BGP を単一プラットフォームで管理可能
- 会場設備からは DHCP でアドレスを取得し、アンダーレイとして使用
- ユーザートラフィックは VPN トンネル経由のデフォルトルート（AD20）で自宅ルーターへ転送

### 上位接続

- 会場設備（管理外 blackbox）から有線で接続。下見時に有線利用の許可を取得済み
- 会場側 AP は使用不可との説明を受けているため、自前で WiFi を提供する必要がある

### 下見時に判明した会場ネットワークの特性

- ゲートウェイ（Alaxala L3 スイッチ）は ICMP に応答しない
- Allied Telesis 製 L2 スイッチ 21 台が RRCP ブロードキャストを常時送出（キャプチャ全体の 38.3%）
- /22 の単一セグメントにスイッチ 29 台が存在し、ブロードキャストドメインが広い
- DNS は会場側サーバ（`10.72.1.11` / `10.72.1.12`）経由で正常動作

### 必要機材

| 機材 | 備考 |
|---|---|
| VyOS ルーター用 PC / VM | 会場ルーター |
| PoE スイッチ | AP・配信 PC 等への給電と VLAN トランク |
| Cisco Aironet 3800（約 20 台） | WiFi 提供（会場 Aruba AP が使えない場合） |
| ローカルサーバ | Grafana / SNMP Exporter / rsyslog / nfcapd |
| ケーブル類 | Cat6 UTP 等 |

## Open Questions / Risks

### 要交渉事項

| # | 項目 | 詳細 |
|---|---|---|
| 1 | **プロキシ解除の確約** | 下見時にプロキシが解除されていなかった実績があるため、当日の解除を確実にする交渉が必要。「解除」の具体的な定義（PAC 配信停止 / 認証無効化 / FW ルール変更）を明確化する |
| 2 | **自前ネットワーク機器の持ち込み許可** | VyOS ルーター、PoE スイッチ、AP 等の持ち込みと有線接続について会場側の許可を取得する |
| 3 | **会場 Aruba AP の利用可否** | VLAN を切ってイベント用臨時 SSID を WLC から配信できるか。可能であれば自前 AP は不要 |
| 4 | **有線ポートの利用範囲** | 下見時は有線利用可とのことだったが、当日も同様か。利用可能なポート数・場所の確認 |

### リスク

| # | リスク | 影響 | 対策 |
|---|---|---|---|
| 1 | 当日もプロキシが解除されない | VPN のアンダーレイが制限される | SoftEther VPN（HTTPS トンネル）で回避。プロキシ認証クレデンシャルがあれば CONNECT 経由で通過可能 |
| 2 | Buffalo ルーターカスケード運用を会場側が推進 | 50〜100 台規模で確実に事故る | 技術的根拠を示して代替案を提示。ただし会場提供機器を受け入れざるを得ない可能性もある |
| 3 | 会場側の協力が得られない | 自前機器の持ち込みや AP 利用が制限される | IPUT の会場をお借りしている立場のため、要望は控えめに。悪手でも受け入れる心構えが必要 |
| 4 | ブロードキャストストーム | 会場の L2 スイッチ 21 台が既にブロードキャストの 38% を占有。当日端末増加で悪化の可能性 | 自前 VLAN で会場側ブロードキャストドメインから隔離 |
| 5 | VPN スループット不足 | 全トラフィックが VPN 経由のため、自宅回線がボトルネックになる可能性 | GCP 経由の経路を primary BGP として併用（オプション） |

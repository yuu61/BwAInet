# NOC 振り返り備忘録: BwAI in Kwansai 2026 (2026-04-19/20)

## 概要

| 項目 | 内容 |
|---|---|
| イベント | Build with AI in Kwansai 2026 (GDG Greater Kwansai 共催) |
| 開催日 | 2026-04-19 / 2026-04-20 |
| NOC 体制 | 経験者 2 名 (user + 先輩) + 初心者 3 名 = 計 5 名 |
| 第一目的 | 動くネットワークの提供 → **達成** |
| 副次目的 | 次世代ネットワークエンジニアの育成・トレーニング → 一定の達成感 |
| 性質 | かなり野良の即席 NOC、ただし二目的とも収穫あり |

## 全体総評

- 動くネットワークの提供という主目的は達成。本番中の障害も都度対応で乗り切った
- 副次目的の育成については、初心者がシリアルコンソールでのスイッチ設定、ケーブル長計算、AP 配置計画など実機に触れる機会を得た
- シニア側 (user・先輩) も RFC や仕様書だけでは見えない、運用してこそ判明するバグ・実装挙動の知見を多く収穫 (本番中の 3 件のインシデント記録参照)
- 一方で、シニアが「当然」と思って言語化しなかった定石が後輩に伝わらず、いくつか手戻り・小トラブルになった (本書の主題)

## 本番中に得られた技術知見 (RFC / 仕様だけでは見えなかった事象)

別ドキュメントに詳細記録あり:

- [GCP DoS 検知 (2026-04-19)](./incident-2026-04-19-gcp-dos-detection.md) — GCP Trust & Safety の DoS 判定はマシンタイプ (egress 上限) に連動しない。capacity 比 1/50 (190 Mbps / 10 Gbps) でも発火。**shaper はパフォーマンス調整ではなくコンプライアンス境界として扱う**
- [IPv6 通信不全 (MLD snooping / RA lifetime)](./incident-2026-04-19-ipv6-mld-snooping.md) — RFC 4541 §3 (link-local scope を MLD snooping 対象外とする推奨) を古い IOS 15 系・Allied Telesis 実装が守らない。**会場利用スイッチは事前に IPv6 multicast 通過試験 (RA / NS) を行うべき**
- [Wi-Fi 瞬断・ローミング失敗・AP 脱落の複合障害](./incident-2026-04-20-wifi-stability.md) — DFS / chan-width / WLC アップリンク冗長化欠如 / AP 世代差 (Wave 1 vs Wave 2) の複合。**Cisco Aironet の型番百位は世代識別子 (3700=Wave1 / 3800=Wave2)、WPA3 等は AP HW 依存で WLC が吸収しない**

これら自体が「運用してこそ得られる現場知見」の典型例で、次回 NOC の一級教材になる。

## 委譲で生じた抜け漏れ —「ベテランが言語化しない定石」

横で見ながら設定を進めたが、シニアが当然と思っている事項を verbalize しないまま渡してしまい、後輩は「目の前の要件に合理的な解」を出した結果、現場の慣例・将来想定・物理ハードの癖・美意識と噛み合わないケースが複数発生した。

整理すると 3 系統に分かれる。

### 1. 設計意図の非伝達 (一番厄介)

成果物を見るまで気付けず、目の前の要件には合理的に見えてしまうため発見が遅れる。

#### 事例: スイッチの幹線 / uplink ポート設計

| 項目 | 内容 |
|---|---|
| 観測した事象 | あるスイッチで幹線/uplink ポートが 1 本しか設定されていなかった |
| シニア側の想定 | 将来スパインに昇格する可能性を見越し、幹線用ポートは複数 (≥2) 確保したかった |
| 後輩側の判断ロジック (推定) | 現状はエッジ用途なので幹線 1 本で十分 (合理的) |
| なぜ複数本にすべきか | 仕様変更で 1 → 複数 にする時、配線・ラベル・config の全部が触り直しになる。最初から複数あれば追加だけで済む |
| 委譲時に必ず言うべきだった一言 | 「このスイッチは将来スパインに昇格する可能性があるので、uplink は 2 本以上で組んで」 |

### 2. 物理ハードウェアの literacy

ラベルは「読む前提」だが、機能を知らないと逆向きでも違和感を持てない類。シニアにとっては自明だが、初心者にとっては「それが大事な情報だと判らない」レベル。

#### 事例: PoE インジェクターの方向逆接続

| 項目 | 内容 |
|---|---|
| 観測した事象 | `data in` / `data & power out` のラベル方向と逆に接続 |
| なぜ重要か | 逆接続だと PoE 給電がスイッチ側に向き、最悪スイッチ側ポートを破損する可能性。少なくとも AP は給電されない |
| 委譲時に必ず言うべきだった一言 | 「PoE インジェクターは方向がある。`data in` がスイッチ側、`data & power out` が AP/カメラ側。ラベル必ず確認」 |
| 同類の罠 | SFP の挿入向き、コンソールのピンアサイン (RJ45 ↔ DB9 / USB)、PoE 規格 (802.3af / at / bt) と AP 世代の対応、AC アダプタの極性 |

#### 事例: 当日構築での配線間違い

| 項目 | 内容 |
|---|---|
| 観測した事象 | 構築フェーズで複数の配線ミス |
| 委譲時に必ず言うべきだった一言 | 「配線前に物理配置図と設計図を一度全部見ろ。差す前に必ずもう一回ラベル確認」 |

### 3. 現場の所作・美意識

「やればすぐ気付くはずの事」だが、所作として身についていないと素通りしてしまう類。

#### 事例: ケーブル長の選定 (50cm vs 3m)

| 項目 | 内容 |
|---|---|
| 観測した事象 | スイッチ - PoE インジェクター間に 50cm の短いパッチが手元にあったのに 3m のケーブルで接続 |
| なぜ重要か | (a) 余ったケーブルがラックの中でループになり見栄えが悪く保守性が下がる (b) 配線追跡時にどこに繋がっているか辿りづらい (c) 在庫を無駄に消費し最終的にケーブル不足の遠因になる |
| 委譲時に必ず言うべきだった一言 | 「ケーブルは長すぎず短すぎず。手元に 50cm があるなら 3m を巻いて使うな。まず周りを見渡してから接続」 |

#### 事例: 配線整理整頓の意識

| 項目 | 内容 |
|---|---|
| 観測した事象 | 配線の見栄えへの意識が低い |
| なぜ重要か | NOC は来賓・スポンサー・参加者がツアーで見る可能性がある。雑な配線 = 雑な運用、と外部から見なされる。技術力の品質シグナル |
| 委譲時に必ず言うべきだった一言 | 「NOC は外から見られる前提。配線の整理整頓は技術力のシグナル。仮配線でも結束しておく」 |

## メタ・パターン: 「委譲時に必ず言う一言」を必須化する

今回浮かび上がった共通構造:

- シニアが「当然」と思っている事項は、verbalize しない限り後輩には伝わらない
- 後輩は能力不足ではなく「目の前の要件に合理的な解」を出してくる
- 問題は、その解が「将来想定 / 物理ハードの癖 / 現場の所作 / 美意識」と噛み合わないこと
- これらの暗黙知は、シニア自身も「言うまでもない」と感じているため、自発的には verbalize されにくい

対策の核: タスクを委譲する時、シニア側に「**定石 / なぜ / 委譲時に必ず言う一言**」の 3 点セットを必ず添える運用にする。最後の「一言」欄を必須項目化することで、暗黙知の verbalize を強制する仕掛けを作る。

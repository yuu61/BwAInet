# Mini PRD: BwAI in Kwansai 2026 Network（最小構成）

> フル構成版は [`PRD.md`](PRD.md) を参照。本ドキュメントは**最低限イベントを成立させる**ための最小構成を定義する。

## 総合所見

**この構成での運用は推奨しない。**

最小構成として成立させるために会場側に依存する要素が多すぎる上、その会場側の対応が信頼できないことが下見で実証されている。具体的には：

1. **プロキシ**: 「解除した」と言われて解除されていなかった前例がある。最小構成はプロキシ解除が前提だが、未解除時にできることが何もない。フル構成であれば VyOS にプロキシ設定を投入するだけで済む
2. **ブロードキャスト**: 会場スイッチ 21 台が L2 ブロードキャストを垂れ流しており、全パケットの 4 割を占有している。Buffalo はブリッジなのでこれを WiFi にそのまま流す。EtherType `0x8899`（L2）のため Buffalo の ACL では弾けない
3. **WiFi**: 家庭用ルーターにはエアタイムフェアネス制御がなく、接続台数が増えると端末あたりのエアタイムが枯渇する。上記ブロードキャストが WiFi 帯域をさらに圧迫する
4. **フォールバックなし**: 上記のいずれかが発生した場合、当日現地で打てる手がない

いずれの問題も、会場側のインフラ設定変更か、自前インフラ（フル構成 [`PRD.md`](PRD.md)）の持ち込みでしか解決できない。最小構成のみで臨んだ場合、当日ネットワークが使えずイベントが成立しないリスクが現実的にある。来場者に有線テザリングを推奨した方がまだマシなレベルである。

## Overview / Problem Statement

BwAI in Kwansai 2026 会場（IPUT）のネットワークはプロキシ認証により外部通信が遮断される可能性が高い（2026-03-24 下見で確認済み）。最小限の機材で参加者にインターネット接続を提供する。

VPN・GCP・VyOS 等の自前インフラは構築しない。会場ネットワークと会場側提供の Buffalo 家庭用 WiFi ルーターのみで運用する。

### 会場条件

- 15 階・16 階の全フロアを貸し切り
- 各教室に RJ45 ポートあり（使用許可済み）
- フロア貸し切りのため、LAN ケーブル等の引き回しも問題ないと考えられる

## Goals

- 参加者が WiFi に接続し、インターネットにアクセスできる
- 会場プロキシの状態に関わらず通信経路を確保する

## Non-Goals

- VPN トンネル（WireGuard / SoftEther）の構築
- GCP / GCE の利用
- VyOS ルーターの構築
- VLAN によるネットワーク分離
- BGP / 経路冗長化
- Grafana / SNMP による監視
- DNS / DHCP の冗長構成

## User Stories

| ユーザー | ストーリー |
|---|---|
| 参加者 | WiFi に接続し、制限なくインターネットにアクセスしたい |
| 配信担当 | YouTube Live への安定したアップリンクを確保したい |

## Proposed Solution

### 通信経路

当日のプロキシ状態に応じて 2 パターンで対応する。

| 条件 | 方式 | 備考 |
|---|---|---|
| プロキシ無効化済み | **会場回線を直接利用** | 追加設定不要。最もシンプル |
| プロキシ無効化不可 | **来場者端末にプロキシ設定を案内** | 各端末で `proxy01.osaka.hal.ac.jp:8080` を設定。クレデンシャルが必要な場合は会場側から発行。HTTP/HTTPS のみ通過可能（ICMP, UDP 系は不可） |

### 構成図

```
[The Internet]
    │
[会場設備 (管理外)]
  DHCP (10.72.1.11 / .12)
  DNS  (10.72.1.11 / .12)
    │
  有線(RJ45) ─┬── 教室RJ45 ─── [Buffalo WiFi ルーター (教室1)] ── WiFi ── [参加者]
              │                   (ブリッジ / L2延伸)
              │
              ├── 教室RJ45 ─── [Buffalo WiFi ルーター (教室2)] ── WiFi ── [参加者]
              │                   (ブリッジ / L2延伸)
              │
              └── 有線 ─── [配信 PC]
```

各教室の Buffalo ルーターはブリッジモード（AP モード）で動作し、会場の有線ネットワークを WiFi で L2 延伸する。DHCP・DNS は会場側の既存インフラをそのまま使用する。

### ネットワーク

- **L2 延伸**: Buffalo ルーターはブリッジとして動作し、会場の有線セグメントをそのまま WiFi に延伸。カスケード（数珠繋ぎ）でもスター（各教室から個別に接続）でも L2 延伸である実態は変わらず、ブロードキャストドメインの共有・プロキシ問題・エアタイム枯渇といった本 PRD で挙げている課題はいずれのトポロジでも同様に発生する
- **DHCP**: 会場側 DHCP サーバ（`10.72.1.11` / `10.72.1.12`）から配布（下見時に正常動作確認済み）
- **DNS**: 会場側 DNS サーバ（`10.72.1.11` / `10.72.1.12`）を使用（下見時に全 345 クエリ正常応答確認済み）
- **セグメント**: 会場側の既存セグメント（下見時は `10.64.56.0/22`）にそのまま参加

### WiFi

各教室に RJ45 ポートが用意されており、使用許可を得ている。会場側が用意する Buffalo 家庭用 WiFi ルーターを各教室の RJ45 に接続し、来場者は各席に近い WiFi に接続する。複数の SSID を設定している場合のローミングや AP 選択はクライアント任せとする。

**使用機器**: 会場側提供の Buffalo 家庭用 WiFi ルーター（ブリッジ / AP モード）

**懸念事項**:
- **エアタイムの枯渇**: 家庭用ルーターは 1 台あたりの無線チャネル（2.4GHz / 5GHz）が限られており、接続台数が増えると端末あたりの送信機会（エアタイム）が激減する。接続自体はできても実効スループットが極端に低下し、通信がタイムアウトする状況が想定される。エンタープライズ AP のようなエアタイムフェアネス制御やバンドステアリングは搭載されていない
- 家庭用ルーターのメーカー推奨同時接続数は 10〜15 台であり、30 台程度が実用上の限界。50〜100 台規模では不安定になる可能性が高い
- EasyMesh は 4 台までの制限がある
- **WLC 不在による負荷偏り**: 1 教室に複数台を設置しても、WLC（無線 LAN コントローラー）がないため端末を各 AP に分散させる制御ができない。端末は最初に接続した AP に留まり続ける傾向があり、特定の AP にだけ接続が集中する
- カスケード接続やメッシュ（air-to-air）は大規模運用に向かない
- **上記を承知の上で、最小構成では会場側提供の機材を使用する**

## Open Questions / Risks

### 最小構成の根本的な限界

**この構成はプロキシ解除が前提であり、プロキシ未解除時のフォールバックを持たない。**

Buffalo ルーターは L2 ブリッジとして動作するため、プロキシをバイパスする機能を持たない。プロキシが解除されなかった場合、この構成単体では対処不可能であり、以下の全てを会場側に依存することになる：

1. **プロキシ認証クレデンシャルの発行** — 会場側が当日用のクレデンシャルを用意してくれるか不明
2. **来場者全員への設定案内** — 各端末で `proxy01.osaka.hal.ac.jp:8080` + クレデンシャルを手動設定する必要がある。OS・ブラウザごとに設定方法が異なり、サポート負荷が極めて大きい
3. **HTTP/HTTPS 以外の通信は全滅** — クレデンシャルがあっても RTMP（YouTube Live 配信）、SSH、QUIC、ping 等は通らない。配信が成立しない可能性がある

2026-03-24 の下見で「プロキシ解除済み」と報告を受けたにも関わらず実際には解除されていなかった前例があり、**当日のプロキシ解除を信頼して最小構成のみで臨むのはリスクが高い。** プロキシ未解除時にイベント自体が成立しない可能性がある。

フル構成（[`PRD.md`](PRD.md)）であれば、プロキシが無効化されていなくても VyOS にプロキシ設定を投入するだけで SoftEther VPN が HTTPS CONNECT 経由でトンネルを確立でき、来場者側の設定変更は一切不要である。最小構成では来場者全員の端末に対して OS・ブラウザごとに異なるプロキシ設定を個別に案内する必要があるのに対し、フル構成では運営側が VyOS 1 台に設定を入れるだけで全来場者の通信が透過的に処理される。これが最小構成との決定的な差であり、プロキシ問題に対する唯一の技術的な解決策である。

### 会場側スイッチの設定問題

下見時のパケットキャプチャ（[解析レポート](iput-pcapng-analysis-report.md)）により、会場側 L2 スイッチに以下の問題が確認されている。

**Allied Telesis 製スイッチ 21 台が RRCP ブロードキャストを常時送出**
- Realtek Remote Control Protocol（RRCP）またはループ検知用の管理フレームを全スイッチが常時ブロードキャスト
- 下見時（端末ほぼ未接続）の時点で**全パケットの 38.3%（5,067 フレーム / 475 秒間）がブロードキャスト**
- レート: 約 10.5 パケット/秒、キャプチャ全期間で均一

**ブロードキャストドメインが過大**
- `/22` の単一セグメントに Allied Telesis 21 台 + HP/Aruba 8 台 = 計 29 台のスイッチが同居
- これらのスイッチが送出するブロードキャストは同一ブロードキャストドメイン上の**全端末が受信・処理する**

**当日の影響**
- 来場者端末が接続されると ARP / mDNS / NBNS / SSDP 等のブロードキャストが加算される
- Buffalo ルーターは L2 ブリッジのため、このブロードキャストドメインにそのまま参加する。会場側スイッチのブロードキャストが WiFi 帯域を消費する
- スイッチ CPU 負荷増大によるフレーム転送遅延やブロードキャストストームのリスク

**この問題は会場側のスイッチ設定（RRCP の無効化、管理 VLAN への隔離、ブロードキャストドメインの分割等）でしか解決できず、最小構成では対処手段がない。**

### その他のリスク

| # | 項目 | 詳細 |
|---|---|---|
| 1 | **プロキシ解除の確約** | 事前に「解除」の具体的内容（PAC 配信停止 / 認証無効化 / FW ルール変更のいずれか）を明確化し、エビデンス（設定変更のスクリーンショット等）を求めるべき |
| 2 | **Buffalo ルーターの同時接続数** | メーカー推奨は 10〜15 台、実用上の限界は 30 台程度。教室あたりの参加者数が多い場合は接続が不安定になる |
| 3 | **ローミング・負荷分散の不在** | WLC がないため、1 教室に複数台の Buffalo を設置しても端末の AP 分散はクライアント任せになる。多くの端末が最初に接続した AP に留まり続けるため、特定の AP に負荷が集中して他の AP は遊ぶという偏りが生じやすい。AP 切替時には一時的な接続断も発生する |
| 4 | **セキュリティ** | 会場の既存セグメント（`10.64.56.0/22`）にフラット参加するため、来場者端末間および会場内の他の端末との通信が素通し |

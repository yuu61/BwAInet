## 4. プロキシ強制環境（通信不可の主原因）

### 4.1 PAC ファイルの配信

PCは `http://10.72.1.61/proxy.pac` から PAC（Proxy Auto-Config）ファイルを取得している（HTTP 200）。

PAC ファイルの要点：

```
デフォルトルール: return PROXY;
プロキシ先:       PROXY proxy01.osaka.hal.ac.jp:8080（10.76.3.62）
```

- DIRECT_LIST に明示されたドメインのみプロキシをバイパス
- DIRECT_LIST には Google、Adobe、Microsoft 365、Unity、Epic/Unreal、Steam 等が登録済み
- **DIRECT_LIST に含まれないドメインは全てプロキシ経由が強制される**

### 4.2 プロキシの認証要求

プロキシ `10.76.3.62:8080` は **HTTP 407 Proxy Authentication Required** を返しており、Basic 認証（realm: "default"）を要求している。

| HTTP レスポンスコード | 回数 | 意味 |
|---|---|---|
| 407 | 33 | プロキシ認証が必要（認証なしでアクセス） |
| 504 | 23 | ゲートウェイタイムアウト（IPv6接続テスト） |
| 200 | 8 | 成功（proxy.pac取得、DIRECT通信） |
| 304 | 6 | キャッシュ有効（変更なし） |

プロキシ認証のクレデンシャルが提供されていないため、プロキシ経由の通信は全て拒否されている。

### 4.3 事前報告との相違

**事前にプロキシの強制は解除済みとの報告を受けていたが、キャプチャ時点では以下の全てが有効のままであった：**

- PAC ファイルの配信（`10.72.1.61`）
- プロキシの Basic 認証（`10.76.3.62:8080`）
- DIRECT_LIST 外ドメインのプロキシ強制

### 5.3 手動コマンドのパケットが存在しない原因

`dig` や `Resolve-DnsName -Server` は Windows DNS Client サービスを経由せず、独自にソケットを開いて DNS クエリを直送する。これらのパケットがキャプチャされていない理由として以下が考えられる：

| コマンド | 推定原因 |
|---|---|
| WSL2 `dig` | WSL2 は Hyper-V VM として動作し、仮想 NIC（vEthernet）経由で通信する。物理イーサネットのキャプチャには映らない |
| PowerShell `Resolve-DnsName -Server` | `-Server` 指定時は DNS Client サービスをバイパスして直接送信するが、Windows Defender ファイアウォール等のアウトバウンドルールでブロックされた可能性がある |

## 11. 総合所見

### 通信不可の原因

```
                        ┌─────────────────────────────┐
                        │   proxy01.osaka.hal.ac.jp    │
  ┌────────┐   PAC 強制  │       10.76.3.62:8080        │
  │ Client ├────────────►│                              │── HTTP 407 ──► 拒否
  │        │             │  Basic認証 realm="default"    │
  └───┬────┘             └─────────────────────────────┘
      │
      │  DIRECT_LIST 該当ドメイン
      └──────────────────────────────────────────────────► インターネット ✅
```

1. **プロキシ認証未解除**: 事前に解除報告を受けていたが、PAC 配信・プロキシ認証・ファイアウォール制御の全てが有効のまま
2. **ICMP フィルタ**: ゲートウェイおよび外向き ICMP が遮断されており、ping による疎通確認が不可能
3. **dig / Resolve-DnsName の失敗**: Windows DNS Client サービスは正常動作しているが、手動コマンドのパケットが物理 NIC から送出されていない（WSL2 仮想 NIC 経由 / ローカル FW による遮断の可能性）
4. **VPN 不通**: UDP 51820（WireGuard）が外向きブロック

### 正常に動作していた部分

- DHCP による IP アドレス取得
- ARP によるゲートウェイ MAC 解決
- Windows DNS Client サービス経由の名前解決（全 345 クエリに応答あり）
- DIRECT_LIST に含まれるドメインへの TLS 通信

## 13. 会場側への確認事項

### 13.1 プロキシ解除の対象と範囲

事前に「プロキシの強制を解除した」との報告を受けていたが、具体的に以下のどれを指していたのか確認が必要。

| 解除の意味として考えられるもの | キャプチャ上の状態 |
|---|---|
| PAC ファイルの配信を停止した | ❌ `10.72.1.61/proxy.pac` が HTTP 200 で配信中 |
| PAC のデフォルトルールを `DIRECT` に変更した | ❌ デフォルトは `return PROXY;` のまま |
| プロキシの Basic 認証を無効化した | ❌ 407 + `Basic realm="default"` が返却されている |
| 特定 IP / セグメントをプロキシ対象から除外した | 不明（クライアント IP `10.64.58.0/22` の除外設定は確認できず） |
| ファイアウォールで直接通信を全許可した | ❌ ICMP・UDP 51820 等が遮断されている |

**確認すべき質問**：「解除」とは具体的にどの機器のどの設定を変更したことを指すのか。変更のエビデンス（設定差分・作業ログ等）はあるか。この下見のために一時的に解除したのか、恒常的な変更だったのか。

### 13.2 過去の通信不具合の有無

今回が初めての事象か、過去にも同様の接続不具合が報告されていたかを確認する。

- 同一セグメント（`10.64.56.0/22`）で以前にも外部通信不可が発生したことがあるか
- 他の利用者（学内ユーザ等）から同様の報告が上がっていないか
- プロキシ認証のクレデンシャルは通常どのように配布されているか（学内アカウント連携 / 個別発行 / 不要のはず、等）

### 13.3 `http_proxy` 等の環境変数設定で通信が通るか

キャプチャから得られた技術的分析は以下の通り。

#### プロキシ経由の CONNECT 試行結果

DIRECT_LIST に含まれないドメインが `proxy01.osaka.hal.ac.jp:8080` に HTTP CONNECT を試みた結果：

| 接続先 | 結果 |
|---|---|
| `api.github.com:443` | **407** (認証要求) |
| `events.gfe.nvidia.com:443` | **407** (認証要求) |
| `filedownload.lenovo.com:443` | **407** (認証要求) |
| `download.wireguard.com:443` | **407** (認証要求) |
| `prod.otel.kaizen.nvidia.com:443` | **407** (認証要求) |

全ての CONNECT リクエストが **407 Basic realm="default"** で拒否されている。これは認証情報が付与されていないためであり、逆に言えば**正しいクレデンシャルを付与すれば通過する可能性がある**。

#### `http_proxy` 設定時の想定動作

```
http_proxy=http://<user>:<pass>@proxy01.osaka.hal.ac.jp:8080
https_proxy=http://<user>:<pass>@proxy01.osaka.hal.ac.jp:8080
```

| 通信種別 | `http_proxy` 設定で通るか | 備考 |
|---|---|---|
| HTTP/HTTPS（プロキシ対応アプリ） | ⭕ 可能性あり | `curl`, `git`, ブラウザ等。Basic 認証のクレデンシャルが正しければ通過する見込み |
| HTTPS CONNECT | ⭕ 可能性あり | `Proxy-Authorization: Basic ...` ヘッダ付きで CONNECT すれば 200 が返る可能性 |
| ping (ICMP) | ❌ 不可 | プロキシは HTTP/HTTPS のみ。ICMP はプロキシを経由できない |
| dig / DNS 直接クエリ | ❌ 不可 | DNS はプロキシ経由にならない。ただし OS の DNS Client 経由は既に動作中 |
| WireGuard (UDP 51820) | ❌ 不可 | UDP トンネルはプロキシを経由できない |
| SSH | ⚠ 条件付き | ProxyCommand で CONNECT を使えば可能な場合がある |
| QUIC (UDP 443) | ❌ 不可 | UDP ベースのためプロキシ不可。HTTP/3 は HTTP/2 にフォールバック |

#### 結論

**クレデンシャルが発行される前提であれば、`http_proxy` / `https_proxy` 設定により HTTP/HTTPS 通信は回復する可能性が高い。** ただし以下の制約が残る：

- プロキシ非対応のプロトコル（ICMP, DNS直接, UDP系）は引き続き使用不可
- プロキシ環境変数を参照しないアプリケーション（一部のデスクトップアプリ等）は個別設定が必要
- WireGuard VPN は利用できないため、VPN 前提の構成は別途検討が必要

**会場側に確認すべき質問**：プロキシ認証のクレデンシャルを当日利用者に発行する運用が可能か。発行可能な場合、認証方式は Basic のままか（NTLM / Kerberos 等への変更予定はないか）。
